## はじめに

AIが生成したテストコードをレビューしていて、`it`ブロックを一つずつ読み解きながら「この関数、結局どういう仕様なんだ？」と感じたことはないでしょうか。

私は普段Claude CodeとCodexを使って開発しており、コードを自分の手で書く時間よりも、AIが生成したコードを読んでレビューする時間のほうが多くなっています。直近のプロジェクトで単体テストの設計を行った際、重要視したのは**人間が仕様を理解するのにかかる認知負荷を最小限にすること**でした。AIが大量のテストケースを生成してくれる時代だからこそ、それを素早く正確に読み解ける構造が必要です。

そこで今回採用したのが、「テーブル駆動テスト」という手法です。本記事では、この手法をVitestで実践する方法を紹介します。

## テーブル駆動テストとは

テーブル駆動テスト（データ駆動テスト、パラメータ化テストとも呼ばれる）は、テストの入力値と期待値をデータの「テーブル（表）」として定義し、同一のテストロジックを各行に対して繰り返し実行する手法です。Go言語のコミュニティで広く使われている手法ですが、言語を問わず適用できます。

この手法には以下のメリットがあります。

- 同じテストロジック（コード）に対して、異なるテストデータセットを繰り返し適用する際に、コードが肥大化しにくい
- 機能変更時に、コードの変更を最小限に抑えることができる
- テスト項目一覧を視認しやすい

## 完成コード

先に、完成品としてのテストコードのサンプルをお見せします。単純な足し算を行う関数の単体テストになっています。

```ts:calc.ts
interface AddParams {
  /** 被加数 */
  augend: number | string;
  /** 加数 */
  addend: number | string;
}
/** 2つの値を数値として加算 */
export const add = ({ augend, addend }: AddParams) => {
  const numericAugend = Number(augend);
  const numericAddend = Number(addend);

  if (Number.isNaN(numericAugend) || Number.isNaN(numericAddend)) return null;
  return numericAugend + numericAddend;
};
```

```ts:calc.test.ts
import { describe, expect, it } from "vitest";
import { add } from "./calc";

/** @see {@link add} */
describe("add", () => {
  const TEST_CASES = [
    {
      title: "正の整数同士を足し算できる",
      args: { augend: 2, addend: 3 },
      expected: 5,
    },
    {
      title: "負の数を含む場合でも正しく計算できる",
      args: { augend: -1, addend: 5 },
      expected: 4,
    },
    // ... 他のテストケースは省略 ...
    {
      title: "数値に変換できない文字列が含まれる場合はnullを返す",
      args: { augend: "abc", addend: 5 },
      expected: null,
    },
  ];

  it.each(TEST_CASES)("$title", ({ args, expected }) => {
    expect(add(args)).toBe(expected);
  });
});

```

## ケースごとにitを書くスタイル

まず、比較対象として、ケースごとに`it`ブロックを書くスタイルのテストコードを見てみます。

```ts
// add.test.ts
import { describe, it, expect } from "vitest";
import { add } from "./add";

describe("add", () => {
  it("正の整数同士を足し算できる", () => {
    expect(add({ augend: 2, addend: 3 })).toBe(5);
  });

  it("負の数を含む場合でも正しく計算できる", () => {
    expect(add({ augend: -1, addend: 5 })).toBe(4);
  });

  it("0を含む場合でも正しく計算できる", () => {
    expect(add({ augend: 0, addend: 7 })).toBe(7);
  });

  it("数値文字列を数値に変換して計算できる", () => {
    expect(add({ augend: "10", addend: "5" })).toBe(15);
  });

  it("片方だけ数値文字列でも計算できる", () => {
    expect(add({ augend: "10", addend: 5 })).toBe(15);
  });

  it("空白付き文字列を変換して計算できる", () => {
    expect(add({ augend: " 10 ", addend: "5" })).toBe(15);
  });

  it("空文字は0として扱われる", () => {
    expect(add({ augend: "", addend: 5 })).toBe(5);
  });

  it("小数を正しく計算できる", () => {
    expect(add({ augend: 1.5, addend: 2.5 })).toBe(4);
  });

  it("指数表記文字列を計算できる", () => {
    expect(add({ augend: "1e2", addend: 5 })).toBe(105);
  });

  it("数値に変換できない文字列が含まれる場合はnullを返す", () => {
    expect(add({ augend: "abc", addend: 5 })).toBeNull();
  });

  it("両方とも変換できない場合はnullを返す", () => {
    expect(add({ augend: "abc", addend: "def" })).toBeNull();
  });
});
```

ちなみにこちらは、ChatGPTに「add関数のテストコードを作成してください」という最小限のプロンプトで作成してもらったテストコードです。

## テーブル駆動テスト

上記のスタイルに対して、私は以下のように考えました。

- 入力値と出力値の対応をもっと見やすくできるのではないか
- expect関数の実行をまとめて行数を減らすことができないか

その結果、冒頭でも紹介したようなテストコードに辿り着きました。
仕様把握に必要な、テストの意図（title）と入力値（args）、出力値（expected）の対応関係のみをテスト実行コードから分離しました。

```ts
import { describe, expect, it } from "vitest";
import { add } from "./calc";

/** @see {@link add} */
describe("add", () => {
  const TEST_CASES = [
    {
      title: "正の整数同士を足し算できる",
      args: { augend: 2, addend: 3 },
      expected: 5,
    },
    {
      title: "負の数を含む場合でも正しく計算できる",
      args: { augend: -1, addend: 5 },
      expected: 4,
    },
    {
      title: "0を含む場合でも正しく計算できる",
      args: { augend: 0, addend: 7 },
      expected: 7,
    },
    {
      title: "数値文字列を数値に変換して計算できる",
      args: { augend: "10", addend: "5" },
      expected: 15,
    },
    {
      title: "片方だけ数値文字列でも計算できる",
      args: { augend: "10", addend: 5 },
      expected: 15,
    },
    {
      title: "空白付き文字列を変換して計算できる",
      args: { augend: " 10 ", addend: "5" },
      expected: 15,
    },
    {
      title: "空文字は0として扱われる",
      args: { augend: "", addend: 5 },
      expected: 5,
    },
    {
      title: "小数を正しく計算できる",
      args: { augend: 1.5, addend: 2.5 },
      expected: 4,
    },
    {
      title: "指数表記文字列を計算できる",
      args: { augend: "1e2", addend: 5 },
      expected: 105,
    },
    {
      title: "数値に変換できない文字列が含まれる場合はnullを返す",
      args: { augend: "abc", addend: 5 },
      expected: null,
    },
    {
      title: "両方とも変換できない場合はnullを返す",
      args: { augend: "abc", addend: "def" },
      expected: null,
    },
  ];

  it.each(TEST_CASES)("$title", ({ args, expected }) => {
    expect(add(args)).toBe(expected);
  });
});
```

ケースごとに`it`を書くスタイルと比較すると、次の違いがあります。

**ケースごとに`it`を書くスタイルでは、「何をテストしているか」と「どうテストしているか」が混在している。** 各`it`ブロックの中に、テストの意図（テスト名）・入力値・期待値・アサーションの実行がすべて含まれており、仕様を把握するにはテストコード全体を読み解く必要があります。

**テーブル駆動テストでは、「何をテストしているか」（TEST_CASES）と「どうテストしているか」（it.each）が分離されている。** テストの意図・入力値・期待値だけがデータとして並ぶため、関数の仕様を把握するにはTEST_CASESだけを読めば十分です。

この分離は、コードレビューの負担も軽減します。仕様の変更や追加があった場合、レビュアーはTEST_CASESの差分だけを確認すればよく、テスト実行ロジックに変更がないことは構造から保証されます。

一方で、アサーションを`toBe(expected)`に統一する構造上、ケースごとに最適なマッチャーを選べないというトレードオフがあります。たとえば、従来のスタイルではnullの検証に`.toBeNull()`を使えますが、テーブル駆動テストでは`.toBe(null)`に統一されます。これにより、テスト失敗時のエラーメッセージの表現力がやや下がる場合があります。とはいえ、テストケースの一覧性と引き換えに許容できる範囲であり、失敗時にはどのケースが落ちたかはタイトルから十分に判断できます。

また、`describe`の上に付けている`/** @see {@link add} */`は、テストファイルからテスト対象の実装へのナビゲーションを目的としたJSDocアノテーションです。IDEの「定義へ移動」機能を使って、テストコードから対象の関数にすぐにジャンプできます。テストを仕様として読む際に、実装の確認が必要になった場面でワンクリックで辿れるようにしておくことで、テストと実装の間の行き来がスムーズになります。

## 適用できる場面・できない場面

テーブル駆動テストは万能ではありません。ここでは、どのような場面に適用でき、どのような場面には向かないのかを整理します。

### 純粋関数の単体テスト → 最適

本記事で紹介しているパターンは、純粋関数の単体テストに最も適しています。「モックや非同期処理を含むケースにも適用できるのか？」という疑問があるかもしれませんが、そもそもそれらは単体テストの責務ではないと考えています。

クリーンアーキテクチャに代表されるレイヤードアーキテクチャでは、一般的にドメイン層（エンティティ、値オブジェクト等）を外部依存のない純粋なロジックとして設計します。このドメイン層にはI/Oやフレームワーク依存が入り込まないため、モックなしでテスト可能であり、テーブル駆動テストが自然に適用できます。

フロントエンド開発で利用されることのある関数型アーキテクチャにも、同様の発想があります。「関数型コア / 命令的シェル（Functional Core, Imperative Shell）」と呼ばれる考え方では、ビジネスロジックを副作用のない純粋な関数として実装し、I/Oや外部依存は命令的シェル（外殻）に押し出します。たとえば、ReactのカスタムHooksから純粋なロジックを別の関数に分離するといった設計です。

https://zenn.dev/loglass/articles/7e40d2a253bfd3

:::message
上記の参考文献では「決定的シェル」と訳されていますが、Gary Bernhardtによる原典の用語は "Imperative Shell" であるため、本記事では「命令的シェル」と表記しています。
:::

いずれのアーキテクチャにも共通するのは、「純粋なロジックを副作用から分離する」という設計方針です。分離された純粋なロジックにモックや非同期処理が入り込むこと自体が設計上の問題であり、そうなっていない純粋なロジックに対して、このパターンは自然に適用できます。

### コンポーネントテスト・結合テスト → 難しい

一方で、コンポーネントテストや結合テストにこのパターンをそのまま適用することは難しいです。これらのテストには共通して、テーブル駆動テストと相性が悪い特徴があります。

- **アサーションが複数種類にわたる** — コンポーネントテストではレンダリング結果の検証、イベント発火後の状態変化、アクセシビリティ属性の確認など、結合テストでは「APIが呼ばれたか」「状態が更新されたか」「エラーハンドリングが正しいか」など、単一の`expected`値では表現しきれないケースが多い
- **テストケースごとに前提条件が異なる** — 結合テストではモックやスタブのセットアップが必要になり、テストケースごとに準備が変わる

無理にテーブル駆動の形に押し込むと、テストケースの定義が肥大化し、本来の目的である可読性の向上とは逆の結果を招きます。

## おわりに

本記事では、テーブル駆動テストを用いて「何をテストしているか」と「どうテストしているか」を分離し、テストケースを仕様として読めるようにする設計を紹介しました。

AIがコードを生成し、人間がそれをレビューするという開発スタイルでは、コードを素早く正確に理解できるかどうかが開発のボトルネックになります。テストコードも例外ではありません。TEST_CASESだけを読めば関数の仕様が把握でき、差分を見るだけでレビューが完結する構造は、このボトルネックを解消する一つの手段です。

実際に、Claude Codeにテストコードを生成させた際、ケースごとに`it`を書くスタイルで出力されることが多く、テストケースの追加漏れや意図の読み取りに時間がかかる場面がありました。従来のスタイルでは、各`it`ブロック内の`expect`まで目を通して「入力は何で、期待値は何か」を一つずつ確認する必要があり、10ケースもあるとそれだけでレビューに時間を取られます。一方、テーブル駆動テストの形式をプロンプトやルールで指定しておくと、TEST_CASESの一覧を上から眺めるだけで「どんな入力パターンが網羅されているか」「期待値は正しいか」が判断でき、レビューが格段に速くなりました。

適用範囲は純粋関数の単体テストに限られますが、関数型コアやドメイン層のように副作用を分離した設計と組み合わせることで、十分に広い範囲をカバーできます。テストコードの「読みやすさ」を設計の対象として意識してみると、開発体験が変わるかもしれません。
